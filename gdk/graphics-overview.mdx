Graphics on Xbox is built on a hardware-tuned variant of Direct3D 12 — **D3D12.x**. It exposes the same shader model and pipeline as desktop D3D12 plus Xbox-specific extensions: monolithic PSO serialization, hardware overlays, native resource management, GPU memory layout control, and the new-feature stack on Xbox Series X|S.

## D3D12.x at a glance

- **Same conceptual API as PC D3D12** — command lists, root signatures, PSOs, descriptors.
- **Native resource management** — you allocate GPU memory directly via `XMemAlloc` and bind it to D3D resources. No driver-side suballocator.
- **Monolithic PSO serialization** — PSOs ship pre-compiled and load instantly. See [Serialized PSOs](#).
- **Tessellation off-chip** — the Xbox One ESRAM-aware off-chip tessellation path is still available where it wins.
- **Frame scheduling and presentation** — explicit present queues, GPU pacing, and hardware overlays for HUD / video.
- **Color and depth compression** — hardware DCC and Cmask. Decode-and-discard paths exposed when you need to read back compressed data.
- **Suspend / resume** — D3D state, command queues, and resources survive PLM suspend; flush rules apply.

## Xbox Series X|S GPU architecture

- **RDNA2-based GPU** with compute units, fixed-function ROPs, and the texture / sampler hardware.
- **GARLIC / ONION memory paths** define which CPU writes are GPU-visible and at what cost. Choose the right path for streaming, vertex buffers, and constants.
- **Texture layouts** — Xbox Series X|S texture layouts are documented and tunable for streaming.
- **Dolby Vision** — supported on Gen9 Xbox Gaming.
- **120Hz output** — supported with display-mode helpers.
- **HDR output** — use [XDisplay](/reference/xdisplay) to negotiate the right HDR mode.

## Differences from PC

Key differences from PC D3D12:

- **No swapchain abstraction** — you present directly into hardware overlay surfaces.
- **No implicit memory residency** — you manage GPU residency explicitly.
- **Different barrier costs** — the Xbox Series X|S RDNA2 cache hierarchy benefits from coarse-grained barriers.
- **PSO serialization** — PSOs are byte-identical across consoles and load instantly.
- **D3D12.x-only extensions** — use the `D3D12_FEATURE_DATA_*` queries to gate on capability rather than `D3D_FEATURE_LEVEL`.

## Tools

- **PIX for Xbox** — capture GPU timing, frame, memory, and shader explorer. See [PIX for Windows](/tools/pix-for-windows) for the PC equivalent.
- **GPU Trace** — long-running timing capture with hardware counters.
- **Mesh and Texture Viewer** — inspect resources at runtime.
- **Xbox Texture Compressor (XBTC)** — offline texture pipeline.
- **Shader Explorer (PIX)** — shader optimization workflows.

## Xbox Series S optimization

Series S shares the Series X feature set but has roughly a third of the GPU compute, less memory, and a lower display target (typically 1440p). Plan for it from day one:

- Target 1440p (or sub-1440p with reconstruction) rather than scaling down a 4K asset budget.
- Use Sampler Feedback Streaming and Variable Rate Shading aggressively — they pay off more here than on Series X.
- Cut shader permutations and effects budget instead of resolution where you can; players notice frame pacing more than pixel count.
- Ship Series S as a first-class build, not a runtime downscale, when feasible.

## Action-safe areas and presentation

Even on flat-panel TVs, players may have overscan or non-standard scaling enabled. Keep critical HUD inside the action-safe area (about 90% of the visible frame), and use the D3D12.x presentation helpers to query the actual output resolution and HDR mode before laying out UI.

## Image processing (WIC)

For non-real-time image work — loading PNG/JPEG title art, generating screenshots, decoding store assets — use **Windows Imaging Component (WIC)**, which is available on Xbox as well as PC. WIC is not for game-asset streaming; use DirectStorage \+ XBTC for that path.
