Xbox consoles expose a unified-memory architecture (UMA) where CPU and GPU share the same physical RAM. The GDK gives your title a fixed budget per console SKU and reserves the remainder for the operating system, capture pipeline, and system overlay. Staying inside that budget — and avoiding fragmentation — is one of the highest-impact things you can do for stability and certification.

<Note>
  Exact memory budgets per console SKU (Xbox One, Xbox One X, Xbox Series S, Xbox Series X) and per running mode (foreground, constrained, streaming) are documented in the NDA-gated [System memory overview](https://learn.microsoft.com/en-us/gaming/gdk/docs/features/console/memory/system-memory-overview). Visit [xbox.com/Developers](https://www.xbox.com/Developers) for program access.
</Note>

## Virtual address space

GDK titles run as 64-bit processes with a large virtual address space. Even so, **commit** is what counts against your budget — reserved virtual address space that is never committed does not consume physical memory. Use this distinction to your advantage: reserve large contiguous ranges for streaming pools and commit/decommit pages as the player moves through the world.

## XMem allocator

The GDK provides `XMemAlloc` and related APIs as a thin, attribute-tagged wrapper around the platform allocator. Tagging each allocation with an `XALLOC_ATTRIBUTES` value lets the platform place pages in the right physical memory pool — for example, write-combined memory for GPU upload buffers or non-cached memory for command lists.

```cpp
#include <XMem.h>

XALLOC_ATTRIBUTES attribs = {};
attribs.s.ObjectType = XALLOC_OBJECT_TYPE_HEAP;
attribs.s.PageSize = XALLOC_PAGESIZE_64KB;
attribs.s.AllocatorId = XALLOC_ALLOCATORID_GAME_MIN;
attribs.s.MemoryType = XALLOC_MEMTYPE_HEAP;

void* buffer = XMemAlloc(1024 * 1024, attribs.dwAttributes);
// ...
XMemFree(buffer, attribs.dwAttributes);
```

Most engines wrap `XMemAlloc` behind their own allocator and apply tags based on the resource being created.

## Budgeting strategies

- **Pre-allocate at boot.** Reserve your largest contiguous heaps during title startup. This avoids fragmentation later in the session.
- **Use streaming pools.** For open-world games, segment textures and meshes into chunks that map to gameplay zones, and stream them in and out of fixed-size pools.
- **Keep a margin.** Leave at least a few hundred megabytes of headroom. Constrained mode, game streaming clients, and certification require a working buffer.
- **Profile early.** Use PIX for Xbox memory captures and the [XBWatson](/tools/service-tools) crash signature service throughout production, not just at cert time.

## Constrained mode

When the player invokes the Xbox guide or a system dialog, your title may be moved into a constrained execution state with reduced CPU, GPU, and memory. Listen for the platform suspend / constrain notifications and shed non-essential work, particularly audio mix complexity and streaming I/O.

## Virtual memory allocator

For large reservations and games that manage their own allocators on top of OS pages, use `XMemVirtualAlloc` / `XMemVirtualFree`. These map directly to the GDK virtual-memory subsystem, support large pages, and let you commit and decommit regions independently — important when your engine pre-reserves a multi-GB address range up front.

## Writeable memory-mapped files

GDK supports writeable memory-mapped files for cases where multiple subsystems (or a tool and your title) need shared access to a backing file with random-access semantics. Prefer memory-mapped files over hand-rolled buffered I/O for save scratch space, on-disk caches, and inter-process scratch data.

## Kernel memory usage

Kernel allocations (handles, file mappings, network buffers, GPU resources) count against your title's budget. Profile kernel usage with the platform memory diagnostics — runaway handle counts or fragmented allocations show up here long before they surface as `E_OUTOFMEMORY`.

## Cross-core memory costs

Accessing memory from a different CPU core than the one that produced it incurs cache-coherency traffic. Lay out producer/consumer data structures by core (per-core arenas, single-writer queues) and avoid pinging shared cache lines across cores during the frame.

## Porting your memory model from XDK

XDK titles that used the Xbox One SDK's allocator can migrate incrementally: keep your custom allocator for hot paths and route everything else through XMem first, then move hot paths over once profiling shows parity. The porting guide in the GDK docs lists the XDK-to-GDK allocator API mappings.
