Social Manager caches every relationship and presence record per local user and surfaces changes as events you process in a `DoWork` loop. This is the recommended way to integrate social state into a running game.

## Lifecycle

| Function | Description |
| --- | --- |
| `XblSocialManagerAddLocalUser` | Begin tracking |
| `XblSocialManagerRemoveLocalUser` | Stop tracking |
| `XblSocialManagerDoWork` | Drain events each frame |
| `XblSocialManagerGetLocalUsers` / `GetLocalUserCount` | Local user enumeration |

## User groups

Filter or fix the cached graph into reusable groups.

| Function | Description |
| --- | --- |
| `XblSocialManagerCreateSocialUserGroupFromFilters` | Filter group |
| `XblSocialManagerCreateSocialUserGroupFromList` | Static list group |
| `XblSocialManagerUpdateSocialUserGroup` | Update a list group |
| `XblSocialManagerDestroySocialUserGroup` | Release |
| `XblSocialManagerUserGroupGetType` / `GetFilters` / `GetLocalUser` / `GetUsers` / `GetUsersTrackedByGroup` | Inspect a group |

## Presence and rich presence

| Function | Description |
| --- | --- |
| `XblSocialManagerSetRichPresencePollingStatus` | Toggle background polling |
| `XblSocialManagerPresenceRecordIsUserPlayingTitle` | Quick check |
