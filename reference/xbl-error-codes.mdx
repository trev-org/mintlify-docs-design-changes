XSAPI returns HRESULTs in the `FACILITY_XBOX` and `FACILITY_HTTP` ranges. Use [`XblGetErrorCondition`](/reference/xsapi-context-and-globals) to map an HRESULT into a coarse condition (`XblErrorCondition`) for retry policy decisions.

## Common condition buckets

| Condition | When you see it |
| --- | --- |
| **Auth** | XBL token expired or refused |
| **Network** | TLS, socket, DNS failures |
| **Rate-limited** | Service throttling |
| **HTTP client** | 4xx — the call is malformed; fix and retry |
| **HTTP server** | 5xx — transient; back off and retry |
| **Generic error** | Other |

## Decoding HRESULTs

- `FACILITY_HTTP` HRESULTs encode the HTTP status code directly: `HRESULT_FROM_WIN32(HTTP_E_STATUS_BAD_REQUEST)`, etc.
- `FACILITY_XBOX` HRESULTs carry XSAPI-specific service errors.

## Diagnosing failures

1. Call `HCHttpCallResponseGetNetworkErrorCode` and `HCHttpCallResponseGetStatusCode` on the failed [HTTP call handle](/reference/xsapi-http-call).
2. If you have an XSAPI HRESULT, run it through `XblGetErrorCondition` for the bucket.
3. Use [`XblHttpCallSetTracing`](/reference/xsapi-http-call) on individual calls to log full request / response data.
