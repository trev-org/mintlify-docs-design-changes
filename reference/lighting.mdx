The GDK dynamic lighting API drives RGB illumination on supported gaming peripherals — keyboards, mice, headsets, controllers — through a single, vendor-neutral surface. Your title pushes color buffers; the platform brokers to whichever connected hardware supports addressable lighting.

For a conceptual walkthrough see [Dynamic Lighting](/gdk/dynamic-lighting).

## Headers and libraries

```cpp
#include <XGameLighting.h>
```

Link against `xgameruntime.lib`.

## Capability negotiation

Before drawing, query each connected device for its capabilities:

- **Lamp count** — how many addressable LEDs the device exposes
- **Lamp position** — 3D coordinates relative to the device frame, for mapping a logical effect to physical LEDs
- **Update mode** — whether the device accepts per-frame updates or only periodic refreshes
- **Color depth** — RGB vs. RGBW vs. monochrome

Query once at startup and again whenever a `device-added` event fires.

## Frame loop

A typical render loop for lighting:

```cpp
// 1. Compute the logical effect for this frame
std::vector<RGBColor> effectBuffer = ComputeLightingEffect(gameState);

// 2. For each compatible device, remap and submit
for (auto& device : compatibleDevices) {
    std::vector<RGBColor> deviceBuffer = RemapToDeviceLamps(
        effectBuffer, device.lampPositions);

    XGameLightingSubmitColorBuffer(device.handle,
        deviceBuffer.data(),
        static_cast<uint32_t>(deviceBuffer.size()));
}
```

Keep the per-device remap fast — lighting runs on the render thread budget.

## Player preference checks

Before pushing color buffers each frame, verify the platform-wide dynamic lighting setting and the per-app override. If either is off, skip submission entirely (don't push black; the OS hands control back to the device's vendor profile).

```cpp
bool enabled = false;
XGameLightingIsEnabledForTitle(&enabled);
if (!enabled) {
    // Player has disabled lighting for this title. Stop submitting.
    return;
}
```

## Suspend handling

When your title is suspended or constrained, stop pushing lighting updates. The OS does not zero the device on suspend by default; if you leave a bright cue running, the player will see it on the dashboard. Push a final "idle" color buffer in your suspend handler.
