PlayFab Party is the cross-platform voice chat and data networking library Microsoft recommends for new GDK titles. It provides a P2P-with-relay transport, built-in voice capture and playback, automatic transcription, and real-time translation — all on a single API surface that ships on Xbox, Windows, Android, iOS, and Switch.

Think of PlayFab Party as the **data plane** companion to PlayFab Lobby / Matchmaking and to [Multiplayer Activity](/services/multiplayer-activity) on the Xbox side.

## Concepts

- **Network** — a logical multiplayer session in Party. Created by one client and joined by others using a network descriptor.
- **Local user** — a signed-in player on this device. Authenticated against PlayFab.
- **Endpoint** — the data sender on a device. One per local user for typical voice \+ data flow.
- **Chat control** — the voice capture / playback object owned by each endpoint.

## Creating and joining a network

```cpp
#include <Party.h>
using namespace Party;

PartyManager& mgr = PartyManager::GetSingleton();
mgr.Initialize(playFabTitleId);

PartyLocalUserHandle localUser;
mgr.CreateLocalUser(playFabEntityId, entityToken, &localUser);

PartyNetworkConfiguration cfg = {};
cfg.maxUserCount    = 8;
cfg.maxDeviceCount  = 8;
cfg.maxEndpointsPerDeviceCount = 1;
cfg.maxDirectPeerConnectionsPerDevice = 7;

PartyNetworkDescriptor descriptor;
mgr.CreateNewNetwork(localUser, &cfg, 0, nullptr, nullptr, nullptr, &descriptor);

// Distribute 'descriptor' (compact binary blob) to other players
// via Multiplayer Activity, PlayFab Lobby, or your own service.
```

Remote players call `ConnectToNetwork` with the descriptor, then `AuthenticateLocalUser` and `ConnectChatControl` to join voice.

## Per-frame work

Each frame:

```cpp
uint32_t count;
PartyStateChangeArray changes;
mgr.StartProcessingStateChanges(&count, &changes);
for (uint32_t i = 0; i < count; ++i) {
    // Dispatch ChatControlCreated, EndpointMessageReceived, etc.
}
mgr.FinishProcessingStateChanges(count, changes);
```

Party drives its own threads under the hood; your job on the game thread is to drain the state-change queue and react.

## Sending data

```cpp
PartyDataBuffer buffer = { payload, payloadSize };
localEndpoint->SendMessage(
    /*targetCount*/ 0,
    /*targets*/ nullptr,                       // null = all remote endpoints
    PartySendMessageOptions::GuaranteedDelivery,
    /*queuingConfig*/ nullptr,
    /*dataBufferCount*/ 1,
    &buffer,
    /*identifier*/ 0);
```

## Voice configuration

Once chat controls are connected, configure listening preferences per peer:

```cpp
localChatControl->SetPermissions(remoteChatControl,
    PartyChatPermissionOptions::SendMicrophoneAudio |
    PartyChatPermissionOptions::ReceiveAudio |
    PartyChatPermissionOptions::ReceiveText);

localChatControl->SetTranscriptionOptions(
    PartyVoiceChatTranscriptionOptions::TranscribeSelf |
    PartyVoiceChatTranscriptionOptions::TranscribeOtherChatControlsWithMatchingLanguages,
    nullptr);
```

## Party vs. Game Chat 2

[Game Chat 2](/services/game-chat) is voice-only and transport-agnostic; Party combines voice and data on a single transport. For new GDK titles that need cross-platform play, **Party is recommended**. Use Game Chat 2 if you already ship on Xbox-only and have an existing voice stack to keep.
