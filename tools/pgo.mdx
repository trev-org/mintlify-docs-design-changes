**Profile-Guided Optimization (PGO)** is a two-build compilation workflow that produces measurably faster Xbox executables — typically 5–15% on hot code paths — at the cost of an extra build step in your release pipeline.

## How PGO works

1. **Instrumented build** — the compiler emits a binary with extra counters at every branch, function entry, and indirect call.
2. **Training run** — you run the instrumented binary against a representative workload (a benchmark scene, an in-game replay, a tutorial). The counters write to a `.pgc` file.
3. **Optimized build** — the compiler re-builds your code, using the `.pgc` data to lay out hot blocks contiguously, inline hot calls, and devirtualize indirect calls.

## When PGO is worth it

- **Shipping retail builds** — always.
- **Performance-critical patches** — yes.
- **Daily dev builds** — no; the overhead is not worth the iteration cost.

## Building a good training scenario

The quality of the training run determines the quality of the optimization. Aim for:

- Cover the title's hottest code paths (combat, traversal, rendering at the target frame rate).
- Avoid loading screens, cutscenes, and menus — they bias the profile toward cold code.
- Use a deterministic in-game replay rather than a human player; otherwise nightly PGO builds will fluctuate.
- Run for several minutes so the counters are statistically meaningful.

## Integrating PGO into CI

A typical pipeline:

```text
build --pgo-instrument  →  deploy  →  run replay  →  collect .pgc
                                                       ↓
build --pgo-use=merged.pgc  →  package  →  ship
```

Keep the training `.pgc` from the previous release as a fallback in case the training run fails for the current build.
